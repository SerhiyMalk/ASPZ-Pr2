# Завдання 2.2
Базова програма: Якщо скомпілювати звичайний "hello world" і запустити size a.out, ми побачимо базові розміри сегментів.
Глобальний неініціалізований масив: Якщо додати int arr[1000]; розмір файлу a.out на диску майже не зміниться, але сегмент bss збільшиться на 4000 байт (1000 * 4 байти).
Глобальний ініціалізований масив: Якщо змінити оголошення на int arr[1000] = {1};, масив переміститься з bss у сегмент data. Це призведе до збільшення розміру сегмента data і фізичного розміру файлу a.out, оскільки ці дані зберігаються у виконуваному файлі.
Локальні масиви: Якщо оголосити ці масиви всередині main(), розміри data та bss не зміняться. Локальні змінні створюються в пам'яті під час виконання програми (у стеку) і не зберігаються в самому a.out.
Оптимізація та налагодження: Компіляція з прапорцем -g  збільшує загальний розмір файлу через додавання символів налагодження, але не впливає на базові сегменти. Використання прапорців оптимізації в першу чергу вплине на зменшення текстового сегмента (text), де зберігаються інструкції.

# Завдання 2.5
У класичній фон-нейманівській архітектурі обійтися без IP неможливо.
Лічильник команд (Instruction Pointer, IP/PC) вказує процесору, де знаходиться наступна інструкція в текстовому сегменті для виконання. Стек же використовується виключно для збереження контексту: локальних змінних, параметрів функцій та адрес повернення.

Вершина стека вказує лише на дані в оперативній пам'яті. Навіть коли процесор читає адресу повернення зі стека , він копіює цю адресу назад у лічильник команд, щоб продовжити виконання. Без регістра IP процесор просто не знатиме, яку інструкцію читати наступною.
